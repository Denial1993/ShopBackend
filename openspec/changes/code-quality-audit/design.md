## 背景說明

ShopBackend 是一個 ASP.NET Core 8 Web API，使用 PostgreSQL（透過 Entity Framework Core + Npgsql）作為資料庫，並整合 Supabase 進行檔案儲存。身份驗證採用 JWT Bearer Token 機制。由於專案目前沒有任何規格文件，很難稽核現有實作是否正確處理邊界條件、安全規則與業務邏輯。

本設計文件說明此次程式碼品質稽核的執行策略。

## 目標與非目標

**目標：**
- 將五大後端功能模組的預期行為記錄為可測試的規格
- 讓未來的開發者（包含程式碼作者本人）能夠驗證實作是否符合預期
- 找出落差 — 即應存在但目前程式碼中缺漏的需求

**非目標：**
- 本次不進行任何執行期程式碼的重構或修改
- 本次不新增自動化測試（這是另一個獨立的變更）
- 本次不涵蓋前端（Vue）行為 — 僅針對後端

## 設計決策

### 決策一：先寫規格，再比對實作

稽核方式採「先定義『應有行為』，再比對程式碼」，而非閱讀程式碼後把現有行為記錄下來。

**理由**：記錄現有行為的風險在於可能將 Bug 寫進規格。以「預期行為」為基準，才能得到真正有效的品質基準線。

**已評估的替代方案**：記錄現有行為 → 否決（有可能掩蓋缺陷）。

### 決策二：每個功能模組一個規格檔

Proposal 中列出的五個功能模組，各自對應一個 `specs/<name>/spec.md` 檔案。

**理由**：保持規格範圍清晰，易於獨立審閱，並與 Controller / Service 的切分邊界一致。

### 決策三：本次不修改 Auth JWT 設定

目前 JWT 實作跳過了 Issuer 與 Audience 驗證（`ValidateIssuer = false`、`ValidateAudience = false`）。這個問題將在規格中標記為已知安全缺口，但本次不予修改。

**理由**：這屬於安全性強化議題，應由一個獨立的專屬變更來處理，以避免本次稽核範圍擴散。

## 風險與取捨

- **【風險】規格可能與現有執行行為衝突** → 緩解方案：在 tasks.md 中將實作落差標記為後續待辦項目，不作為本次稽核的阻塞項。
- **【風險】ECPay 整合在沒有真實測試環境下難以完整驗證** → 緩解方案：規格僅涵蓋已知的回呼契約，不確定之處標記為待確認問題。
- **【風險】可能遺漏部分驗證邏輯的涵蓋** → 緩解方案：交互比對 `Validators/` 資料夾與各規格的驗證需求。
